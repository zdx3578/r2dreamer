
# ARC3 规则增强世界模型：阶段性文档 v0.1
日期：2026-03-17

## 0. 本轮确认的状态合同（按你的简写收紧）

统一世界状态写成：

`W_t = <X_t, H_t, R_t, G_t, E_t, A_t>`

其中：

- `X_t`：显式规范状态（explicit canonical state）。是真实世界状态的主承载。
- `H_t`：小型 latent / memory context。只负责隐藏模式、长时依赖和局部歧义。
- `R_t`：Rule state / RuleBeliefState。维护候选规则、已验证规则、竞争规则簇、上下文守卫、例外关联和规则置信。
- `G_t`：frontier / search graph。维护已探索状态—动作图、frontier、tested-actions、回路与回访路径。
- `E_t`：exception / residual state。保存解释失败、规则冲突、长期未解释变化、负对照失败与异常簇。
- `A_t`：audit chain。保存规则提出、验证、反证、降权、退役证据，以及关键实验轨迹。

## 1. 一个必须固定下来的约束

**以上六项都保存记忆。以上六项都保存记忆。以上六项都保存记忆。**

但它们的记忆类型不同：

### 1.1 快速工作记忆
- `X_t`
- `H_t`

用途：当前步决策、短时滚动更新。

### 1.2 规则/异常记忆
- `R_t`
- `E_t`

用途：规则后验、竞争关系、例外、残差升级。

### 1.3 搜索记忆
- `G_t`

用途：避免重复试错、维护 frontier、支持动作效率。

### 1.4 证据记忆
- `A_t`

用途：审计、复盘、sign-off、失败归因、规则 promotion / retirement。

> 结论：这不是“只有规则存记忆”，而是 `X/H/R/G/E/A` 全部是可持久化状态，只是写频率、保留窗口、跨 episode 继承策略不同。

## 2. ARC3 版本的总原则

### 原则 1：ARC3 采用 explicit-state-first，而不是 latent-first
ARC3 官方环境是 turn-based 的 2D grid 系统，输入是 1-N 帧 JSON state + metadata，grid 最大 64x64，值域 0-15，并返回可用动作集合。对这种任务，显式规范状态 `X_t` 必须是主状态，`H_t` 只能做上下文补充，而不能替代主状态。  
来源：[ARC-AGI-3 Games Docs], [ARC-AGI Toolkit Quickstart], [ARC-AGI-3 page]

### 原则 2：世界模型优先预测“变化”和“机制”，不是优先重建整帧
ARC3 的评分核心是完成度 + 动作效率，且内部推理不计动作，环境动作才计费。因此第一性目标是减少无效动作、定位有效变化、压缩试错，而不是把整帧压进大 latent 后再猜机制。  
来源：[ARC-AGI-3 Methodology]

### 原则 3：动作空间是动态 schema，不是固定动作头
ARC3 有标准动作接口，目前文档中包含 ACTION1-7，并明确每步会返回 available actions；其中 ACTION6 只告诉你可用，不会给出活跃坐标。  
来源：[ARC-AGI-3 Actions Docs]

### 原则 4：搜索图 `G_t` 是世界模型状态的一部分
公开的 ARC3 图探索方法已经说明：维护已探索状态—动作图、frontier、tested-action 跟踪，本身就是强先验，不应该只是外挂搜索器。  
来源：[Graph-Based Exploration for ARC-AGI-3], [arc-agi-3-just-explore]

### 原则 5：`R_t` 不是 rules list，而是 rule belief state
必须显式维护：
- candidate rules
- validated rules
- competing clusters
- contextual guards
- exceptions link
- confidence / support
- open validation tasks

### 原则 6：`E_t` 是一等公民，不是噪声桶
长期未解释变化、规则互相打架、负对照失败、重复假阳性，都进入 `E_t`，不能被默默塞回 latent。

### 原则 7：`A_t` 是工程必须品，不是后期 embellishment
ARC3 本地开发可以高吞吐离线运行，在线/scorecard 则提供回放与官方计分。因此审计链要从第一版就定义，不要等“模型强了再补”。  
来源：[Local vs Online Docs], [Recordings & Replays], [ARC Harness]

### 原则 8：E 层只做“解释框架体检”
E 层不是 reward head。它负责判断当前解释框架是否越来越误导系统，是否该提高实验预算、降低某规则置信、升级异常簇。E 层的自然位置在 `R_t/E_t/A_t` 上方的 summary diagnostic，而不是动作价值头。  
来源：[Deeply Felt Affect], [Deep temporal models and active inference], [pymdp]

## 3. 新主线：5 个运行时模块 + 2 个协议

## 模块 M1：CanonicalStateAdapter
输入：ARC3 observation / frame chain / metadata / available actions  
输出：`X_t`

职责：
- 把 JSON state 转成稳定、规范、可 diff 的显式状态
- 产出 deterministic object / component / region / channel views
- 产出 state hash、delta-ready snapshots

说明：
- 这里可以借鉴旧 ARC repo 的 observation contract / frame digest / connected components 经验
- 但不要再做“地图构建 + 假设管理 + 策略选择”一锅炖

## 模块 M2：Transition Core
输入：`X_t, H_t, a`
输出：`ΔX_t`, `Δevent_t`, `H_{t+1}`

职责：
- 先预测“动作是否导致变化”
- 再预测“变化发生在哪里、作用到哪个对象/通道、属于什么 effect family”
- 不强求整帧生成式重建

建议分两个子头：
- `ChangePredictor`
- `DeltaPredictor`

## 模块 M3：Rule Belief Engine
输入：历史 `Δevent`、上下文、已知规则证据  
输出：`R_t`, 局部 rule-conditioned predictions

职责：
- 候选机制聚类
- 规则置信更新
- 竞争簇管理
- 守卫与例外维护
- 输出实验价值线索

## 模块 M4：Frontier Graph Planner
输入：`X_t, R_t, G_t, E_t`
输出：下一步动作

职责：
- 管理 frontier
- 优先测试高信息量、低动作浪费动作
- 兼顾 progress gain 与 rule disambiguation gain

ARC3 评分是 Relative Human Action Efficiency（RHAE），每关分数按 `(human_baseline_actions / ai_actions)^2` 计，且后期关卡权重更大，所以 planner 必须显式计入动作成本。  
来源：[ARC-AGI-3 Methodology]

## 模块 M5：Audit + E Diagnostic
输入：规则候选、实验轨迹、负对照、异常簇  
输出：`A_t` 与 E-level summary

职责：
- 规则提出与反证链
- promotion / retirement reason
- exception escalation
- E 层体检面板

### 协议 P1：Rule Promotion / Retirement
候选规则进入 validated 规则前必须过：
- 一致性
- 专属性
- 时序性
- 上下文守卫
- 干预复现
- 负对照

### 协议 P2：Residual Escalation
对高频未解释变化：
- 升级到 `E_t`
- 触发候选机制生成
- 提高对应动作的实验收益
- 必要时降低旧规则置信

## 4. 因果接口要不要提前实现？

### 结论
**要提前定义接口，但不要一开始全部放开。**

不是“先把所有因果流程端到端实现”，而是：

- 先把因果/反事实/实验设计的**接口合同**定好
- 然后按阶段逐步打开功能
- 早期先走 observational + audit
- 中期再走 intervention + refutation
- 后期再走 counterfactual + query-driven experiment design

### 为什么要前置接口
原因不是学术完整性，而是工程代价。

如果现在不预留这些接口，后面要把规则系统升级成因果系统时，会出现三类破坏性返工：

1. `R_t` 只会存“规则句子”，不会存 query / intervention / evidence schema
2. `A_t` 没有反证和负对照结构，后面没法做 causal refutation
3. planner 没有“信息增益”或“区分竞争规则”的动作价值接口，后面很难接实验设计

### 但为什么不能一次放开
ARC3 当前最硬的瓶颈仍是：
- 状态规范化
- 动作变化建模
- frontier 搜索
- action efficiency

这从官方规则、图探索基线和 StochasticGoose 这类方法里都看得很清楚：先把“哪些动作会改变世界”学稳、把 state-action frontier 管理稳，收益最大。  
来源：[ARC-AGI-3 Methodology], [Graph-Based Exploration for ARC-AGI-3], [StochasticGoose]

### 因果接口的渐进开放顺序

#### Phase 0：先定义 schema，不启用复杂推理
预留：
- `CausalQuery`
- `InterventionSpec`
- `CounterfactualQuery`
- `RefutationReport`
- `RuleEvidencePacket`
- `ExperimentProposal`

此时都可以是 lightweight dataclass / protocol。

#### Phase 1：只启用 observational causal hooks
启用：
- 基于 `Δevent` 的候选规则生成
- 支持/反支持统计
- guard / exception 记录
- 简单负对照
- rule posterior 更新

不启用：
- 显式 do-calculus
- counterfactual rollout
- Bayesian BOED planner

#### Phase 2：启用 intervention hooks
启用：
- “针对竞争规则，哪个动作最能区分它们” 的实验价值评分
- 明确记录 intervention outcome
- 规则 promotion 时要求干预复现证据

#### Phase 3：启用 query-driven experiment design
启用：
- 对指定 causal query 计算 expected information gain
- 有目标地选择实验动作，而不是只做全图盲搜

#### Phase 4：启用 counterfactual / refutation shell
启用：
- under-rule simulation
- 负对照、假阳性追踪
- counterfactual discrepancy report

## 5. 贝叶斯、自由能、主动推理，应该怎么接？

### 5.1 贝叶斯推理：现在就该进来
最适合立刻进入 ARC3 主线的，是“规则信念的贝叶斯化”，不是整套自由能训练。

#### 该放在哪里
- `R_t`：规则后验、竞争簇、守卫、例外
- `E_t`：异常后验、未解释变化簇
- `A_t`：证据似然与反证记录
- planner：实验动作的预期信息收益

#### 具体形式
把每条候选规则看成假设 `r_i`，维护：
- prior
- posterior
- support likelihood
- contradiction likelihood
- context-conditioned validity

这一步不需要引入完整 PPL 训练，只需要有清晰的 belief update 接口。

### 5.2 自由能 / Active Inference：先放在 E 层与实验价值，不放在底层地图与动作模型
主动推理强调深层生成模型、分层时间结构和 epistemic behaviour；`pymdp` 也明确把 expected free energy 与 epistemic value 作为离散状态规划的核心。  
来源：[Deep temporal models and active inference], [pymdp]

但对 ARC3 当前阶段，最合理的位置不是：
- 用自由能替代基础动作模型
- 用 EFE 直接替代整个 planner

更合理的位置是：

#### 位置 A：E 层诊断
把“解释误差是否下降、规则置信是否更稳、例外是否在扩散”整合成 model-fitness diagnostic。

#### 位置 B：实验价值项
把 epistemic value / expected information gain 作为 planner 的一个分项：
- 区分两条竞争规则值不值
- 这个动作能不能减少规则后验熵
- 这个动作是不是只会制造噪声

#### 位置 C：多时间尺度证据整合
active inference / deep temporal models 的一个强点，是把证据积累放在嵌套时间尺度上。这很适合以后把 `X/H/R/G/E/A` 的证据更新做成多时间尺度，而不是只看一步。  
来源：[Deep temporal models and active inference]

### 5.3 E 层的正确定位
Hesp 等人关于 valence 的工作更支持这样一种放置方式：  
valence 更像 beliefs-about-beliefs 的二阶 summary，而不是外部 reward 本身。  
来源：[Deeply Felt Affect]

因此：
- 不把 E 层接成 reward head
- 不让 E 层主导基础动作预测
- 只让 E 层读 `R_t/E_t/A_t`，输出“当前解释框架是否在误导系统”的 summary

## 6. 外部论文 / 开源方法：现在最值得借什么

## A. 立即采用 / 直接接入

### A1. 官方 ARC-AGI Toolkit + Docs
必须对齐官方接口，而不是围绕旧 fork 再发明新契约。  
价值：
- local/offline 开发
- official action schema
- scorecards / replays
- competition mode 约束  
来源：[ARC-AGI Toolkit Quickstart], [Local vs Online], [Recordings & Replays], [Methodology], [Actions]

### A2. ARC Harness `arcagi3`
适合作为本地 benchmark / orchestration / experiment harness。  
来源：[ARC Harness]

### A3. `arc-agi-3-just-explore`
这是非常强的结构先验参考，不是要照搬，而是要吸收它的两点：
- graph node = canonical frame / state hash
- frontier-driven action selection  
来源：[Graph-Based Exploration for ARC-AGI-3], [arc-agi-3-just-explore]

### A4. StochasticGoose
价值不在“一个完整方法”，而在第一阶段的动作建模形态：
- 先预测动作会不会导致 frame change
- 再分层选 action / coordinate  
来源：[StochasticGoose]

## B. 现在预留接口，但不直接塞进主循环

### B1. DoWhy
DoWhy 明确提供四步因果推理接口，并强调 assumptions + validation + refutation。它非常适合借来设计 `A_t` 的 refutation / validation protocol，但不适合在线主控制环。  
来源：[DoWhy GitHub], [DoWhy Docs]

### B2. causal-learn
适合做离线规则关系分析、时间片 causal discovery 和对 `R_t` 里的竞争簇做辅助分析，但不适合在线每步调用。  
来源：[causal-learn GitHub]

### B3. Pyro
Pyro 很适合承载：
- 规则后验原型
- 变分推断
- Bayesian optimal experimental design（BOED）  
但这应是 notebook / offline analysis / small subsystem 级别，而不是现在就接在线 agent 核心。  
来源：[Pyro Docs], [Pyro OED Docs]

### B4. PyMC / CausalPy
PyMC 现在已经有 do-operator 示例，CausalPy 也把 Bayesian SCM 明确成 graph + likelihood + priors。它们更适合作为“离线反事实分析”和“实验报告工具”，不是在线 planner。  
来源：[PyMC do-operator], [CausalPy SCM]

### B5. ABCI
ABCI 的价值非常高：它把 causal discovery、causal reasoning 和 experimental design 放在同一个 fully-Bayesian loop 里，而且有开源实现。  
但它的计算图比较重，不适合现在直接在线集成。更适合作为：
- `ExperimentProposal` 设计参考
- `R_t` 的 query-oriented belief update 参考
- 后续 notebook 原型  
来源：[ABCI paper], [ABCI GitHub]

### B6. CausalWorld
CausalWorld 最适合作为“验证壳 benchmark”。它公开支持 do-interventions、counterfactual environments、structured observations 和 curriculum through interventions。等我们把 ARC3 上的规则系统跑通后，很适合拿它做 intervention / refutation 流程验证。  
来源：[CausalWorld GitHub]

### B7. CausalARC
CausalARC 不是 ARC3 的直接实现参考，但它很重要，因为它把 ARC 风格推理任务直接建立在 SCM 上，而且提供 static dataset 与随机任务生成代码。它更适合作为“规则—因果—反事实评价协议”的思想来源。  
来源：[CausalARC], [CausalARC site]

### B8. pymdp
pymdp 不适合直接做 ARC3 主体，但它是一个很好的“E 层与 epistemic value”原型工具。  
来源：[pymdp GitHub]

## 7. 对旧 r2dreamer / 旧 ARC repo 的工程借鉴边界

## 可以借
### 来自 r2dreamer
- 训练入口与 runner 外壳
- 配置体系
- 日志、checkpoint、实验目录约定
- replay / eval / sign-off 的工程组织习惯

### 来自旧 ARC repo
- frame digest / transition record / JSONL trace
- failure taxonomy
- component / object / hierarchy 的实用经验
- 调试脚本与可视化风格

## 不能继续沿用为 ARC3 核心
### 不建议延续
- Dreamer latent 作为 ARC3 主状态
- slot objectification 作为 ARC3 主表示
- `rho-only shadow rollout` 路径作为 ARC3 主循环
- map builder / hypothesis bank / policy / audit 强耦合
- active inference / EFE 直接做控制主核

> 总判断：**工程复用外壳，算法重写核心。**

## 8. 下一阶段实现顺序（建议）

### 阶段 S0：先把接口合同写死
实现 dataclass / protocol：
- `CanonicalState`
- `HiddenContext`
- `RuleBeliefState`
- `FrontierGraph`
- `ExceptionState`
- `AuditChain`
- `DeltaEvent`
- `RuleCandidate`
- `InterventionSpec`
- `RefutationReport`

### 阶段 S1：跑通最小 ARC3 baseline
只做：
- `X_t`
- `G_t`
- ChangePredictor
- action / coordinate proposal
- frontier planner

目标：
- 少做 no-op
- 有稳定 state hash
- 有可回放实验记录

### 阶段 S2：接 `R_t`
只做：
- 候选规则生成
- 支持/反支持
- 竞争簇
- guard / exception 记录

不做：
- counterfactual
- Pyro / PyMC 复杂推断
- active inference planner

### 阶段 S3：接 `E_t + A_t`
只做：
- residual cluster
- negative controls
- rule promotion / retirement
- audit-first sign-off

### 阶段 S4：再开放 Bayesian experiment hooks
只做：
- expected information gain
- query-driven experiment proposal
- competing rules disambiguation bonus

### 阶段 S5：最后才谈自由能 / active inference 强整合
只把它接到：
- E 层诊断
- 多时间尺度 belief update
- 实验价值归纳

## 9. 这份阶段性文档的最终判断

### 判断 1
ARC3 的规则增强世界模型，不该再实现成“Dreamer 主干上叠很多解释层”。

### 判断 2
更合适的实现是：
**显式状态转移系统 + 小型历史上下文 + 规则信念状态 + frontier 图 + 异常状态 + 审计链。**

### 判断 3
因果接口需要现在就定义，但只应该缓慢放开。

### 判断 4
贝叶斯方法现在就该进入 `R_t / E_t / A_t`；  
自由能 / active inference 现在不该接管基础动作模型，而应先作为：
- E 层诊断
- 实验价值分项
- 多时间尺度证据整合原则
